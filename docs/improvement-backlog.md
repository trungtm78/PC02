**PROBLEM_STATEMENT:**
`apiFetch` helper function is defined locally inside both `CaseObjectsTab.tsx` and `LawyerListPage.tsx` (and similar page components), duplicating the same authenticated-fetch pattern across page-level components.

**ROOT_CAUSE:**
Each page component was implemented in isolation. The `apiFetch` pattern (attach JWT from localStorage, set Content-Type, handle 401) is a cross-cutting concern that was deferred to avoid scope creep during TASK-2026-261225.

**IMPACT:**
- Any change to auth token handling (e.g., token refresh, header name change) must be applied in multiple files manually.
- Increases regression risk when auth flow is modified in the future.
- Estimated duplication: ~15–20 lines × N page files.

**RECOMMENDATION:**
- **SHORT_TERM:** Extract to `frontend/src/lib/apiFetch.ts` — a shared module exporting `apiFetch(url, options?)` that reads token from localStorage and sets Authorization header.
- **LONG_TERM:** Integrate with a proper API client layer (e.g., Axios instance with interceptors) when the project adopts a centralized API client pattern.

**EFFORT:** S (2–4 hours)
**PRIORITY:** P2 — not blocking, but will compound as more pages are added

**ACCEPTANCE_CRITERIA (when resolved):**
- `apiFetch` defined exactly once in `frontend/src/lib/apiFetch.ts`
- All page components import from that shared utility
- No local `apiFetch` definitions remain in page files
- All existing E2E and UAT tests continue to pass after refactor
